**Важно:** эти данные обновляются **ежеквартально** (сплит-форма) и при пересмотре методик (774/812-пр — редко). Значит, в отличие от ГЭСН/ФСБЦ (которые меняются с выходом очередного «доп.» раз в несколько месяцев), сплит-форму нужно перезагружать каждый квартал перед началом работы над новыми сметами — заложено в схему как версионируемый импорт (см. `price_periods` в разделе 3).

#### Структура Сплит-формы (лист «Сплит-форма», данные с 21-й строки, 9 колонок)

| # | Колонка | Смысл |
|---|---|---|
| 1 | Код ресурса, услуги | Совпадает 1-в-1 с кодами `materials`/`machines`/labor-кодами в ГЭСН |
| 2 | Наименование | — |
| 3 | Единица измерения | `чел.-ч` = труд, `маш.-ч` = машина, прочее = материал/услуга |
| 4 | Отпускная цена на 01.01.2022 | = `OptCost` в ФСБЦ |
| 5 | Сметная цена на 01.01.2022 | = `Cost` в ФСБЦ (базовая цена) |
| 6 | № группы однородных ресурсов (ГОСР) | Группа для индексации (напр. 1030 «Бульдозеры») |
| 7 | Наименование группы ГОСР | — |
| 8 | Сметная цена в текущем уровне цен | Заполнена НАПРЯМУЮ для позиций без индекса (услуги перевозки, труд) |
| 9 | Индекс изменения сметной стоимости к группе ГОСР | Заполнен, когда текущая цена считается через индекс: `текущая = цена_01.01.2022 × индекс` |

Правило пересчёта цены в текущий уровень: **если колонка 8 непустая (≠ '-') — берём её как есть; иначе — `цена(кол.5) × индекс(кол.9)`.** Оба варианта встречаются в одном файле (машины чаще идут через индекс, но не всегда).

Три категории строк по коду ресурса:
- **Материалы/оборудование** (~230 тыс. строк, коды как в `ФСБЦ_Мат_Оборуд.xml`) — цена через индекс группы ГОСР.
- **Машины** (1 732 строки, коды `91.*`, полностью совпадают с `ФСБЦ_Маш.xml`) — цена машино-часа (без з/п машиниста) через индекс группы ГОСР.
- **Транспортные/грузовые услуги** (коды `2xxxxx-*`, `4xxxxx-*` — авто- и ж/д перевозка, ПРР) — готовая текущая цена в колонке 8, без индекса.

**Пробел №1 закрыт полностью**: 186 строк с единицей `чел.-ч` — это и есть тарифные ставки труда, с кодами, совпадающими 1-в-1 с кодами ресурсов внутри `Work/Resources` в файлах ГЭСН:

| Префикс кода | Категория | Пример |
|---|---|---|
| `1-100-XX` | Рабочие-строители, средний разряд X.X (71 строка, разряд 1.0–8.0 с шагом 0.1) | `1-100-38` → «Средний разряд работы 3,8» |
| `2-100-XX` | Рабочий N разряда (8 строк, целые разряды 1–8) | `2-100-05` → «Рабочий 5 разряда» |
| `3-100-XX` | Техник I/II/III категории | `3-100-02` → «Техник II категории» |
| `3-200-XX` | Инженер I/II/III категории | `3-200-01` → «Инженер I категории» |
| `3-300`, `3-400` | Ведущий инженер, Главный технолог | — |
| `4-100-XX` | Машинисты, средний разряд X.X (91 строка) | `4-100-050` → «Средний разряд машинистов 5,0» |
| `5-100-XX` | Работник-исполнитель реставрационных работ | — |

Готовая ставка (руб/чел.-ч на 2 кв. 2026, Москва) — прямо в колонке 8, никакой доп. индексации не требуется.

**Важное открытие про машины**: в `ФСБЦ_Маш.xml` у каждой машины есть атрибут `DriverCode` (например, `DriverCode="4-100-050"`) — это прямая ссылка на тарифный код машиниста из сплит-формы. То есть з/п машиниста внутри маш.-часа не берётся из статичного поля `SalaryMach` (это старое справочное значение на 01.01.2022), а пересчитывается КАЖДЫЙ квартал как `labor_tariff_rates[DriverCode].rate × LabourMach`. Проверено на выборке — совпадает 1-в-1.

**Пробел №2 закрыт**: см. раздел 4.6 — таблица `work_type_norms`, собранная из приказов 774/пр и 812/пр (135 строк, включая подпункты), с приоритетной привязкой к `base_type` + `collection_code` (коду Сборника).

**Пробел №3 закрыт**: индекс из сплит-формы (колонка 9) — это и есть региональный/квартальный индекс пересчёта, но не единый на всю смету, а **свой для каждой группы ГОСР** (пыль/бетон/металл и т.д. дорожают по-разному). Прежняя идея одной таблицы `regional_indices` с одним числом на регион+период была неверной — индекс всегда идёт через `Код ресурса → Группа ГОСР → Индекс`, то есть уже встроен в саму сплит-форму построчно, отдельная таблица не нужна.

#### Открытый вопрос, требующий проверки (не решается только по данным XML)

В нормах ГЭСН встречаются **два способа кодирования труда рабочих**, они взаимоисключающие в пределах одной расценки (проверено на всех 29 976 нормах `ГЭСН.xml` — пересечений нет):
- **78% работ**: разряд явно указан — код вида `1-100-38` с `EndName="Средний разряд работы 3,8"`. Расценивается напрямую по тарифу из сплит-формы.
- **~10% работ (3 128 из 29 976)**: код труда указан «голым» — `Resource Code="1"` без разряда и без цены.

Аналогично с трудом машинистов: код `2` (без разряда) встречается **всегда и только** вместе с хотя бы одним ресурсом-машиной (`91.*`) — ни разу не отдельно. При этом код `4-XXX-YY` (разряд машиниста) ни разу не встречается напрямую как `Resource` внутри `Work` — только внутри атрибута `DriverCode` у машины. Это сильно намекает, что «голый» код `2` — это, по сути, справочное количество чел.-ч машинистов, уже учтённое в цене маш.-часа через `DriverCode`, и его **не нужно** тарифицировать второй раз (иначе будет задвоение).

Для «голого» кода `1` (рабочие без разряда, без привязки к машине) такой подсказки нет — что именно он означает (сумма в рублях на 01.01.2022, требующая только индексации, или чел.-ч по некоему «умолчательному» разряду) однозначно не определяется по имеющимся файлам. Рекомендация: свериться с текстовой/технической частью соответствующих сборников ГЭСН (обычно идёт приложением к сборнику, не входит в XML-выгрузку) или с Методикой определения сметной стоимости (Приказ Минстроя № 421/пр, раздел про ресурсный метод). До уточнения — в MVP такие позиции (~10%) помечать флагом «требует уточнения тарифа» вместо того, чтобы молча считать их неверно.

## 2. Технологический стек (рекомендация)

- **СУБД: SQLite**, файл `gesn2022.db`. Обоснование: данные — справочные, read-heavy, ~80 тыс. норм + 47 тыс. цен прекрасно помещаются в SQLite (ожидаемый размер БД с индексами — не больше 300–400 МБ), не нужен отдельный сервер, файл легко переносить/версионировать, деплоить рядом с любым проектом (в т.ч. встраивать в RPKM/Bitrix24 как источник данных через отдельный микросервис или прямое чтение).
- **Импорт-скрипт: Node.js**, т.к. это разовый ETL под существующий стек (сервер cv7451529, pm2). Библиотеки: `sax` или `saxes` (потоковый парсинг для `ГЭСН.xml` и `ГЭСНм.xml` — они большие, DOM-парсинг лучше не использовать) + `better-sqlite3` для записи.
- Если удобнее на Python — логика идентична, `lxml.etree.iterparse` + `sqlite3`. Выбор не принципиален, ниже даю схему и маппинг независимо от языка.

## 3. Схема БД

```sql
-- Единая таблица норм по всем 5 базам ГЭСН
CREATE TABLE works (
  id INTEGER PRIMARY KEY,
  base_type TEXT NOT NULL,        -- 'ГЭСН' | 'ГЭСНм' | 'ГЭСНмр' | 'ГЭСНп' | 'ГЭСНр'
  code TEXT NOT NULL,             -- напр. '01-01-001-01'
  collection_code TEXT,           -- код Сборника
  collection_name TEXT,
  section_path TEXT,              -- 'Раздел > Подраздел > Таблица' текстом, для поиска/отображения
  name_full TEXT NOT NULL,        -- NameGroup.BeginName + Work.EndName, склеенные
  measure_unit TEXT,
  content_text TEXT,              -- состав работ (Content/Item.Text через \n), справочно
  nr_code TEXT,                   -- код пункта приказа НР из NrSp/ReasonItem/@Nr, напр. 'Пр/812-012.0'
  sp_code TEXT,                   -- код пункта приказа СП из NrSp/ReasonItem/@Sp, напр. 'Пр/774-012.0'
  UNIQUE(base_type, code)
);
CREATE INDEX idx_works_code ON works(code);
CREATE VIRTUAL TABLE works_fts USING fts5(name_full, content=works, content_rowid=id); -- полнотекстовый поиск по названию

-- Нормы расхода ресурсов на единицу работы
CREATE TABLE work_resources (
  id INTEGER PRIMARY KEY,
  work_id INTEGER NOT NULL REFERENCES works(id),
  resource_code TEXT NOT NULL,
  resource_type TEXT NOT NULL,    -- 'labor' | 'machine' | 'material' | 'equipment' | 'abstract_material'
  quantity REAL NOT NULL,         -- норма расхода на ед. измерения работы (для abstract_material — коэффициент расхода основного материала)
  end_name TEXT,                  -- напр. 'Средний разряд работы 3,8' (для labor — важно для подбора тарифа)
  measure_unit TEXT,
  is_abstract INTEGER DEFAULT 0,  -- 1 = источник <AbstractResource> («неучтённый»/основной материал: код ГРУППЫ КСР, конкретную марку выбирает пользователь)
  tg_codes TEXT                   -- @TechnologyGroups абстрактного ресурса (через ';') — из каких технологических групп выбирать конкретный ФСБЦ-код
);
CREATE INDEX idx_wr_work ON work_resources(work_id);
CREATE INDEX idx_wr_resource ON work_resources(resource_code);

-- Материалы и оборудование (из ФСБЦ_Мат_Оборуд.xml)
CREATE TABLE materials (
  code TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  category TEXT,                  -- 'Материал' | 'Оборудование'
  measure_unit TEXT,
  cost REAL,                      -- розничная цена
  opt_cost REAL                   -- оптовая цена
);

-- Машины и механизмы (из ФСБЦ_Маш.xml)
CREATE TABLE machines (
  code TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  measure_unit TEXT,
  salary_mach REAL,               -- зарплата машиниста в составе маш.-часа
  labour_mach REAL,
  price_cost_without_salary REAL, -- стоимость маш.-часа без зарплаты
  with_relocation INTEGER,        -- bool
  driver_code TEXT,
  machinist_category REAL,
  electricity REAL,
  electricity_cost REAL
);

-- Технологические группы (взаимозаменяемые материалы)
CREATE TABLE technology_groups (
  tg_code TEXT NOT NULL,
  resource_code TEXT NOT NULL,    -- материал, входящий в группу замены
  PRIMARY KEY (tg_code, resource_code)
);

-- Привязка норм к технологическим группам (какой абстрактный ресурс в норме заменяем)
CREATE TABLE tg_work_links (
  tg_code TEXT NOT NULL,
  work_code TEXT NOT NULL,
  base_type TEXT NOT NULL,
  abstract_resource_code TEXT NOT NULL,
  abstract_resource_name TEXT,
  measure_unit TEXT
);
CREATE INDEX idx_tgwl_work ON tg_work_links(base_type, work_code);

-- Поправочные коэффициенты (справочно, подбор вручную)
CREATE TABLE coefficients (
  code TEXT PRIMARY KEY,
  section_path TEXT,
  name TEXT,
  target TEXT,                    -- 'Расход' и т.п.
  decree_use TEXT,
  machine_value REAL,
  material_value REAL,
  labor_costs_value REAL,
  machinist_labor_costs_value REAL
);

-- Периоды сплит-формы (версионирование — тарифы/индексы обновляются ЕЖЕКВАРТАЛЬНО)
CREATE TABLE price_periods (
  id INTEGER PRIMARY KEY,
  region TEXT NOT NULL,           -- напр. 'город Москва'
  year INTEGER NOT NULL,
  quarter INTEGER NOT NULL,       -- 1..4
  source_letter TEXT,             -- реквизиты письма Минстроя (справочно)
  imported_at TEXT NOT NULL,
  UNIQUE(region, year, quarter)
);

-- Тарифные ставки труда (из сплит-формы, строки с MeasureUnit='чел.-ч')
CREATE TABLE labor_tariff_rates (
  period_id INTEGER NOT NULL REFERENCES price_periods(id),
  resource_code TEXT NOT NULL,    -- '1-100-38', '2-100-05', '3-100-02', '4-100-050' и т.п. — точное совпадение с work_resources.resource_code / machines.driver_code
  name TEXT,                      -- 'Средний разряд работы 3,8'
  rate_per_hour REAL NOT NULL,    -- готовая ставка в текущем уровне цен периода
  PRIMARY KEY (period_id, resource_code)
);

-- Цены/индексы материалов и машин по группам ГОСР (из сплит-формы, все остальные строки)
CREATE TABLE price_period_resources (
  period_id INTEGER NOT NULL REFERENCES price_periods(id),
  resource_code TEXT NOT NULL,    -- совпадает с materials.code / machines.code / кодом услуги перевозки
  gosr_group_no INTEGER,          -- № группы однородных строительных ресурсов
  gosr_group_name TEXT,
  base_price REAL,                -- цена на 01.01.2022 (колонка 5, если применимо)
  current_price REAL,             -- готовая текущая цена (колонка 8), если задана напрямую
  index_value REAL,               -- индекс к группе ГОСР (колонка 9), если цена считается через индекс
  PRIMARY KEY (period_id, resource_code)
);
CREATE INDEX idx_ppr_period ON price_period_resources(period_id);

-- Нормативы НР (812/пр) и СП (774/пр) по видам работ — см. раздел 4.6
CREATE TABLE work_type_norms (
  id INTEGER PRIMARY KEY,
  item_no TEXT NOT NULL,          -- '1.1', '6.2', '104доп' и т.п. — номер строки в приложении к приказу
  work_type_name TEXT NOT NULL,
  section TEXT,                   -- 'I'..'VI' (раздел таблицы приказа)
  base_type TEXT,                 -- 'ГЭСН'|'ГЭСНм'|'ГЭСНп'|'ГЭСНр'|'ГЭСНмр'|NULL (для раздела VI "Прочие работы")
  collection_codes TEXT,          -- через ';', код(ы) Сборника, либо 'ALL' для ГЭСНп
  sbornik_note TEXT,              -- пояснение из приказа (что именно за сборники/разделы/отделы)
  profit_pct REAL,                -- норматив сметной прибыли, % от ФОТ (774/пр)
  overhead_territory_pct REAL,    -- норматив НР для Территории РФ, % от ФОТ (812/пр)
  overhead_mprks_pct REAL,        -- норматив НР для МПРКС
  overhead_rks_pct REAL,          -- норматив НР для РКС
  needs_manual_review INTEGER     -- 1 = внутри Сборника несколько строк норматива различаются по Разделу/Таблице/условиям — авто-подбор по одному base_type+collection_code неоднозначен, нужна ручная проверка
);
```

## 4. Маппинг импорта (файл → таблица)

### 4.1 `ГЭСН.xml`, `ГЭСНм.xml`, `ГЭСНмр.xml`, `ГЭСНп.xml`, `ГЭСНр.xml` → `works` + `work_resources`

Обход дерева `ResourceCategory > Section (вложенный, N уровней) > NameGroup > Work`:
- `base_type` — из атрибута корня `<base BaseType="...">`.
- `code` — `Work/@Code`.
- `section_path` — конкатенация `Section/@Name` всех уровней вложенности над текущим `Work` через ` > `.
- `collection_code`/`collection_name` — первый уровень `Section` (`Type="Сборник"`).
- `name_full` — `NameGroup/@BeginName` + `" "` + `Work/@EndName`.
- `measure_unit` — `Work/@MeasureUnit`.
- `content_text` — все `Content/Item/@Text` через `\n` (может отсутствовать — `<Content />`).
- Для каждого `Resource` внутри `Work/Resources`:
  - `resource_code` = `Resource/@Code`
  - `quantity` = `Resource/@Quantity`
  - `end_name` = `Resource/@EndName` (может отсутствовать)
  - `measure_unit` = `Resource/@MeasureUnit` (может отсутствовать)
  - `resource_type` — определяется **после** загрузки справочников `materials`/`machines` (см. 4.6), не в момент парсинга норм: сначала загрузить `materials` и `machines`, затем при импорте `work_resources` смотреть, есть ли `resource_code` в `machines` → `machine`; иначе в `materials` (по `category`) → `material`/`equipment`; иначе → `labor`.
- **`AbstractResource` внутри `Work/Resources` (КРИТИЧНО, легко пропустить)** — это «неучтённый»/основной материал: `Code` — код ГРУППЫ КСР (формат `XX.X.XX.XX`, без конечного `-XXXX`), `Quantity` — коэффициент расхода на единицу нормы, `TechnologyGroups` — код(ы) технологической группы. Импортировать в `work_resources` с `resource_type='abstract_material'`, `is_abstract=1`, `tg_codes=@TechnologyGroups`. При расчёте пользователь выбирает конкретный материал: `technology_groups[tg_code]` → список кодов ФСБЦ → выпадающий список. Пример: норма 12-01-015-03 содержит `<AbstractResource Code="01.2.03.03" Name="Мастики битумосодержащие" MeasureUnit="т" Quantity="0.05" TechnologyGroups="27.03.039"/>` — в смете Гранд-Сметы это строка «Н», под которую добавляется подпозиция с выбранным ФСБЦ-кодом.
- **`NrSp/ReasonItem` — ОБЯЗАТЕЛЬНО импортировать**: `@Nr` → `works.nr_code`, `@Sp` → `works.sp_code`. Это прямая привязка каждой нормы к пункту приказа 812/пр (НР) и 774/пр (СП) — тот же код виден в выгрузке Гранд-Сметы («Пр/812-012.0-1 НР Кровли»). Благодаря этому подбор норматива НР/СП становится детерминированным по самой норме, без эвристик по сборникам.

### 4.2 `ФСБЦ_Мат_Оборуд.xml` → `materials`

Обход `ResourceCategory[@Type] > Section (вложенный) > Resource`:
- `code` = `Resource/@Code`, `name` = `Resource/@Name`, `measure_unit` = `Resource/@MeasureUnit`
- `category` = ближайший `ResourceCategory/@Type` (`Материал` | `Оборудование`)
- `cost` = `Prices/Price/@Cost`, `opt_cost` = `Prices/Price/@OptCost`

### 4.3 `ФСБЦ_Маш.xml` → `machines`

Обход `ResourceCategory > Section (вложенный) > Resource`:
- `code`, `name`, `measure_unit` — аналогично
- из `Prices/Price/@*`: `salary_mach` ← `SalaryMach`, `labour_mach` ← `LabourMach`, `price_cost_without_salary` ← `PriceCostWithoutSalary`, `with_relocation` ← `WithRelocation`, `driver_code` ← `DriverCode`, `machinist_category` ← `MachinistCategory`
- из `ExpendableMaterials/Material/@*`: `electricity`, `electricity_cost`

### 4.4 `Сплит-форма_<регион>_на_<N>_квартал_<год>.xlsx` → `price_periods` + `labor_tariff_rates` + `price_period_resources`

Лист «Сплит-форма», данные с 21-й строки (строка 19 — заголовки, строка 20 — номера колонок 1–9). Для каждой строки данных:
- `resource_code` = колонка 1, `name` = колонка 2, `unit` = колонка 3
- Если `unit == 'чел.-ч'` → строка идёт в `labor_tariff_rates`: `rate_per_hour` = колонка 8 (это готовая ставка, `'-'` здесь не встречается).
- Иначе → строка идёт в `price_period_resources`: `gosr_group_no` = колонка 6 (если не `'-'`), `gosr_group_name` = колонка 7, `base_price` = колонка 5 (если не `'-'`), `current_price` = колонка 8 (если не `'-'`), `index_value` = колонка 9 (если не `'-'`).
- `period_id` — создать/найти запись в `price_periods` по региону+году+кварталу (парсится из имени файла или из ячеек B4/B5 листа — «Наименование субъекта» и «Наименование зоны»).
- Файл большой (281 тыс. строк, ~14 МБ) — читать через `openpyxl` в режиме `read_only=True`, без загрузки всего файла в память построчно.

Формула получения актуальной цены ресурса на нужный период:
```
current_price = row.current_price if row.current_price is not None
                else row.base_price * row.index_value
```

### 4.5 Приказы 774/пр и 812/пр → `work_type_norms`

Таблицы нормативов сметной прибыли (774/пр, 1 колонка %) и накладных расходов (812/пр, 3 колонки % — Территория/МПРКС/РКС) объединены построчно по номеру пункта/подпункта таблицы (нумерация видов работ у обоих приказов идентична). Готовый разбор — файл `work_type_norms.csv` (135 строк, включая подпункты), приложен к этому ТЗ.

**Подбор норматива — детерминированный, через `works.nr_code` / `works.sp_code`** (см. 4.1): каждая норма несёт коды вида `Пр/812-012.0` и `Пр/774-012.0`, где числовая часть после дефиса = номер пункта таблицы приказа (`012.0` → пункт 12 «Кровли»). В `work_type_norms.csv` колонка `item_no` соответствует этой нумерации — при импорте нормализовать оба формата к единому ключу (например `812:12.0`, `774:21.1`) и матчить строго по нему. Прежняя эвристика по `base_type` + `collection_code` остаётся только как fallback для валидации импорта (расхождение ключей = сигнал об ошибке транскрипции CSV). Флаг `needs_manual_review` при таком подборе в основном теряет смысл — оставить только для случаев, когда `nr_code`/`sp_code` в норме отсутствует.

**✅ Актуализация редакций выполнена.** `work_type_norms.csv` (136 строк) пересобран из редакций приказов **с изменениями**: 812/пр — изм. от 02.09.2021 и 26.07.2022 (приложение в редакции, действующей с 12.09.2022), 774/пр — аналогичная действующая редакция. Отличий от исходных редакций 2020 г. — 7 пунктов (автодороги: НР 126→147, СП 95→134; мосты и трубы: НР 122→140, СП 80→93; плюс НР по пяти «атомным» пунктам 6.2, 7.2, 9.1, 49.1, 55), добавлен новый п.110 «Перевозка работников автомобильным транспортом» (НР 100/100/100, СП 88). Верификация по эталонной ЛСР Гранд-Сметы: п.21 (147/134), п.11 (112/65), п.12 (109/57) — совпадение полное. Если Минстрой снова изменит приказы — процедура обновления та же: распарсить таблицу приложения, обновить CSV, прогнать diff.

### 4.6 Порядок импорта (важно)

1. `materials` ← `ФСБЦ_Мат_Оборуд.xml`
2. `machines` ← `ФСБЦ_Маш.xml`
3. `works` + `work_resources` ← пять файлов ГЭСН* (с классификацией `resource_type` по уже загруженным справочникам)
4. `technology_groups` ← `База_ТГ.xml` (обход `TechnologyGroup/@Code` → список `Resource/@Code`)
5. `tg_work_links` ← `Ключи_перехода_ТГ.xml` (обход `TechnologyGroup/@Code > Work[@Code,@BaseType] > AbstractResource[@Code,@Name,@MeasureUnit]`)
6. `coefficients` ← файл поправочных коэффициентов (обход `Section` вложенный, `section_path` аналогично п.4.1, `Coefficient/@code,@target,@decreeuse` + вложенные `Name`, `Machine/Value`, `Material/Value`, `LaborCosts/Value`, `MachinistLaborCosts/Value`)
7. `price_periods` + `labor_tariff_rates` + `price_period_resources` ← сплит-форма (загружается **при появлении новой сплит-формы**, то есть ежеквартально — независимо от шагов 1–6, которые выполняются один раз при обновлении самого ГЭСН/ФСБЦ)
8. `work_type_norms` ← `work_type_norms.csv` (разово, обновляется только если Минстрой пересмотрит методики 774/812-пр)

## 5. Логика расчёта сметы

Вход: список позиций `{ base_type, work_code, quantity, period_id, territory_type, coefficient_codes?[] }`, где `territory_type` ∈ {Территория, МПРКС, РКС} — влияет только на НР.

Для каждой позиции:

1. `work = SELECT * FROM works WHERE base_type=? AND code=?`
2. `resources = SELECT * FROM work_resources WHERE work_id = work.id`
3. Для каждого ресурса считаем `line_cost` и одновременно копим `ФОТ_рабочих` и `ФОТ_машинистов` (нужны для НР/СП, см. ниже):
   - **`material`/`equipment`**: `price = price_period_resources[period_id, resource_code].current_price` (см. формулу расчёта актуальной цены в 4.4); `line_cost = resource.quantity * position.quantity * price`.
   - **`machine`**: два слагаемых —
     - без-з/п-часть: `machine_price = price_period_resources[period_id, machine.code].current_price` (это и есть `price_cost_without_salary`, пересчитанный по индексу группы ГОСР);
     - з/п машиниста: `driver_rate = labor_tariff_rates[period_id, machine.driver_code].rate_per_hour`; `salary_part = driver_rate * machine.labour_mach`;
     - `line_cost = resource.quantity * position.quantity * (machine_price + salary_part)`;
     - `ФОТ_машинистов += resource.quantity * position.quantity * salary_part`.
   - **`labor`, код с разрядом (`1-100-XX`, `2-100-XX`, `3-*`, `5-100-XX`)**: `rate = labor_tariff_rates[period_id, resource.resource_code].rate_per_hour`; `line_cost = resource.quantity * position.quantity * rate`; `ФОТ_рабочих += line_cost`.
   - **`labor`, «голый» код `2`**: по имеющимся данным зарплата машиниста уже учтена через ресурсы-машины той же расценки (см. открытый вопрос в разделе 1) — **не тарифицировать повторно**, использовать только как справочное количество чел.-ч (для отчётности/проверки), `line_cost = 0`.
   - **`labor`, «голый» код `1`**: однозначного правила нет (открытый вопрос) — считать `line_cost = NULL`, помечать позицию флагом `требует_уточнения_тарифа = true` вместо того, чтобы тихо занижать смету.
   - **`abstract_material` (основной/неучтённый материал)**: пользователь выбирает конкретный код ФСБЦ из технологической группы (`tg_codes` → `technology_groups` → список кандидатов) либо указывает произвольный код. `price` — по выбранному коду через сплит-форму (базисная × индекс группы ГОСР); `line_cost = resource.quantity * position.quantity * price`. В выводе показывать отдельной подпозицией (как «1.1.1 ФСБЦ-...» в ЛСР Гранд-Сметы). Пока материал не выбран — позиция считается без него с явной пометкой «основной материал не выбран».
4. Если для позиции указаны `coefficient_codes` — применить `Machine/Material/LaborCosts/MachinistLaborCosts` множители из `coefficients` к соответствующим суммам статей (выбор коэффициента — вручную пользователем, система только показывает подходящие по `section_path` варианты и подставляет).
5. Прямые затраты (ПЗ) по позиции = Σ материалы + Σ машины + Σ труд.
6. **НР и СП** (по формулам из 774/пр и 812/пр):
   - `ФОТ_i = ФОТ_рабочих + ФОТ_машинистов` (п.14 Методики 812/пр, п.14 Методики 774/пр — формула идентична в обоих приказах).
   - Норматив — детерминированно по `work.nr_code` и `work.sp_code` (см. 4.5).
   - `НР_i = ФОТ_i * norm(nr_code).overhead_[territory_type]_pct / 100`
   - `СП_i = ФОТ_i * norm(sp_code).profit_pct / 100`
   - Если `nr_code`/`sp_code` у нормы отсутствует или не найден в справочнике — не считать автоматически, вывести позицию с пометкой «выберите норматив вручную» и списком кандидатов по сборнику.
7. Итог по позиции = ПЗ + НР + СП. Итог по смете = Σ по всем позициям.

## 6. MVP-объём для первой итерации

1. Импорт-скрипт (Node.js, потоковый парсинг) → наполнение SQLite по разделу 4 (шаги 1–6, разово) + отдельный импортёр сплит-формы (шаг 7, повторяемый ежеквартально) + загрузка `work_type_norms.csv` (шаг 8, разово).
2. CLI или простой HTTP-эндпоинт: поиск норм по названию (FTS) и по коду.
3. Функция расчёта позиции сметы (раздел 5) для случая «код труда с явным разрядом» (78% работ) — покрывает основную массу расчётов уже в MVP, поскольку тариф, индексы и НР/SП теперь полностью определены реальными файлами.
4. Явный флаг «требует уточнения тарифа» для ~10% работ с «голым» кодом `1` — не блокирует остальной расчёт, но не даёт тихо занизить смету.
5. Коэффициенты и `needs_manual_review`-нормативы — везде, где авто-подбор неоднозначен, отдавать выбор пользователю, а не гадать.

Технологические группы (ТГ) — второй приоритет, не блокируют базовый расчёт, нужны для точности подбора материалов на нетиповых работах.

## 7. Продуктовый сценарий (v1 — персональный инструмент)

Пользователь v1 — сам владелец (сметчик-эксперт). Основной кейс: **быстро получить нормативную стоимость работы в текущих ценах и сравнить с рыночной ценой из КП подрядчика**. Инструмент НЕ заменяет Гранд-Смету: если по итогам сравнения нужно юридическое обоснование, полная смета делается в Гранд-Смете по номерам расценок, которые выдал инструмент. Отсюда требования:

- **Старт сессии расчёта — выбор/загрузка сплит-формы.** Первый экран нового расчёта: список уже загруженных периодов (`price_periods`) + кнопка «Загрузить новую сплит-форму» (xlsx → импортёр из 4.4 → новый период). Каждая смета/расчёт жёстко привязывается к `period_id`, на котором считалась — при загрузке следующего квартала старые расчёты не пересчитываются молча, но по ним доступна кнопка «пересчитать в ценах периода X».
- **Вход**: расценка ГЭСН (поиск по коду или названию через FTS) + объём + выбор основного материала из технологической группы (если в норме есть `abstract_material`) + опционально рыночная цена за единицу для сравнения. Переключатели: НДС (×1.2), коэффициенты к норме (например «увеличение толщины ×4» — множитель к расходам ресурсов), территория (для НР).
- **Выход**: развёрнутая расценка со структурой ОТ / ЭМ / ОТм / М / основной материал / ФОТ / НР / СП / итого — за единицу измерения нормы (100 м² и т.п.), за «человеческую» единицу (руб/м²) и на объём. Если задана рыночная цена — рядом дельта в рублях и процентах (сравнение только итогов, без постатейного матчинга). Обязательно выводить **номера использованных расценок и кодов материалов** — они и есть мост в Гранд-Смету.
- Экспорт в Excel — упрощённая табличная форма расчёта (не обязателен формат Приложения 3 к 421/пр в v1, достаточно читаемой структуры с кодами).

## 8. Верификация движка

Эталон — реальная ЛСР из Гранд-Сметы («Монолит 2025, Москва», РИМ по Методике 2020, 3 072 строки). **Важно: эталон составлен в ценах III кв. 2024, а загруженная сплит-форма — II кв. 2026, поэтому текущие цены совпадать НЕ будут и НЕ должны.** Сверка по слоям, не зависящим от периода:

1. **Ресурсная часть** — чел.-ч, маш.-ч, расходы материалов с учётом объёма и коэффициентов к норме: должна совпадать с эталоном до последнего знака (это чистая арифметика норма × объём × коэффициент).
2. **Базисные цены на 01.01.2022** (колонка 8 эталона) — одинаковы в любой сплит-форме, должны совпадать.
3. **Проценты НР/СП и их коды** («Пр/812-…», «Пр/774-…») — должны совпадать с эталоном (с поправкой на актуализацию редакций, см. 4.5).
4. Полная сверка текущих цен до рубля возможна, только если загрузить сплит-форму именно III кв. 2024 — опционально, если файл сохранился.

Тест «расчёт позиции 1.1 (ГЭСН27-04-001-01) и 1.4 (ГЭСН12-01-015-03) воспроизводит эталон по слоям 1–3» — обязательный acceptance-критерий MVP.
