# ТЗ Этап 5: AI-ассистент подбора расценок и проверки смет

> Дополнение к `ТЗ_смета_ГЭСН2022.md`. Этапы 0–4 закрыты и являются фундаментом:
> БД `gesn2022`, движок расчёта позиции, поиск (стемминг/опечатки/синонимы),
> веб-интерфейс, экспорт. Этот этап добавляет AI-слой поверх, ничего в фундаменте
> не переписывая.

## 1. Цель

Два пользовательских сценария, оба стартуют с загрузки xlsx-ведомости в свободной
форме (как `Гранит.xlsx` — КП подрядчика, не ГРАНД-формат):

**Сценарий А — «Составить смету»**: по ведомости работ подобрать нормы ГЭСН,
собрать черновик сметы с расчётом через существующий движок.

**Сценарий Б — «Проверить смету подрядчика»**: по КП с ценами подобрать нормы,
посчитать нормативную стоимость и показать сравнение с ценами подрядчика
построчно + итогом.

Оба сценария выдают **черновик на подтверждение человеку**, не финальную смету.
Каждая позиция несёт обоснование и степень уверенности. Смету подписывает
сметчик; ассистент экономит время на подборе, а не заменяет ответственность.

## 2. Ключевое архитектурное изменение

Инструмент перестаёт быть офлайновым: AI-подбор работает через внешний API
(серия tool-use вызовов с сервера). Всё остальное (поиск, расчёт, экспорт,
ручной режим) продолжает работать без сети как раньше — AI-слой строго аддитивен.

Провайдер у заказчика — **neuroapi.host**: OpenAI-совместимый агрегатор
(перепродаёт GPT/Claude/Gemini через единый эндпоинт формата OpenAI).
Из этого следует:

- SDK: **`openai` для Node** с кастомным `baseURL` (эндпоинт neuroapi;
  точный URL взять из их документации/личного кабинета — обычно вида
  `https://neuroapi.host/v1`). НЕ `@anthropic-ai/sdk` — он с этим эндпоинтом
  не совместим.
- Конфиг через `.env` (в `.gitignore`): `AI_API_KEY`, `AI_BASE_URL`,
  `AI_MODEL`. Ключ в коде, коммитах и логах не появляется. Без ключа —
  AI-функции задизейблены с пояснением, остальное работает.
- Выбор модели: нужна модель с надёжным function calling. Стартовое
  предложение — актуальная Claude Sonnet или GPT-класса модель из их списка
  (что доступно и дешевле); имя в конфиге, смена модели не требует правок кода.
- **Шаг 0, обязательный до всей остальной работы: smoke-тест tool use через
  прокси.** Агрегаторы транслируют function calling между форматами провайдеров,
  и для не-OpenAI моделей это иногда работает криво (теряются tool_calls,
  ломается parallel calling). Написать минимальный скрипт: один вызов с одной
  простой tool-функцией, проверить, что модель возвращает корректный
  структурированный tool_call и принимает tool-результат. Если для выбранной
  модели через прокси tool use нестабилен — сменить модель на ту, где стабилен,
  и зафиксировать это в отчёте. Не строить пайплайн на непроверенном звене.
- Fallback-режим на случай, если tool use через прокси не работает ни на одной
  доступной модели: пайплайн без native tool use — модель просит поиск обычным
  текстом в согласованном JSON-формате, сервер выполняет и возвращает результат
  следующим сообщением. Медленнее и хрупче, но рабоче; реализовывать только
  если smoke-тест провалится.
- Считать и показывать расход токенов на прогон (usage из ответов суммируется,
  вывести в UI по завершении).
- **Стоимость измерять по разнице баланса кабинета neuroapi до и после прогона,
  а не по счётчикам токенов.** Установлено на шаге 0: usage у прокси
  недостоверен и врёт по-разному в зависимости от модели (у одной модели
  4 399 prompt-токенов на короткую фразу, у другой `cached_tokens` больше
  самого `prompt_tokens`). Поэтому в отчёте контрольной точки — баланс до,
  баланс после, разница в рублях; счётчики токенов приводить справочно.

## 3. Компонент 1: Парсер ведомости (без AI)

Вход: xlsx в свободной форме. Задача — извлечь структуру позиций до передачи
модели. Наблюдаемая структура на примере `Гранит.xlsx` (2 листа):

- строка заголовков (`№ п/п`, `Наименование`, `Ед. изм.`, `Кол-во`,
  `За единицу`, `Всего`) — искать по содержимому, номер строки не фиксирован;
- секции (`Работы`, `Материалы`, `УСН`) — строка с номером без единиц/цен;
- позиции — номер вида `1.1`, наименование, ед., кол-во, цены;
- итоги (`Итого:`, `Всего по разделу:`) — не позиции;
- листов может быть несколько — каждый парсится отдельно.

Выход парсера: JSON `{sheet, sections: [{name, items: [{no, name, unit, qty,
unit_price, total}]}], totals}`. Эвристики неизбежно неполные — если структура
листа не распозналась (нет строки заголовков), честно вернуть ошибку с просьбой
показать файл, а не тихо пропустить лист.

Парсер отдельным модулем `src/ai/parse-vedomost.js` + юнит-тесты на
`Гранит.xlsx` (оба листа: 14+12 позиций работ/материалов на листе «Гранит»,
10+12 на листе «Электрика» — проверить точные числа по факту).

## 4. Компонент 2: Пайплайн подбора (AI + tools)

Один прогон ведомости = серия вызовов модели с tool use. Модели даются
инструменты (реализованные как функции поверх существующего кода):

1. `search_norms(query, base_type?)` — существующий `/api/search` (топ-20:
   код, название, ед., сборник, раздел).
2. `get_norm_details(base_type, code)` — существующий `/api/work`: полный
   состав ресурсов, Content (состав работ), ед. изм., абстрактные ресурсы
   с кандидатами, НР/СП.
3. `search_materials(query)` — поиск по таблице materials (для сценария Б:
   сопоставление материалов КП с ФСБЦ). Нужен новый простой эндпоинт
   (LIKE/FTS по name) — единственный новый серверный примитив.

Системный промт модели (хранить в `src/ai/prompts/`, не инлайном в коде):
- роль: инженер-сметчик, подбирающий нормы ГЭСН-2022 к ведомости;
- контекст: весь JSON ведомости целиком (чтобы позиция «монтаж ступеней»
  читалась в контексте «смета на гранит», а не изолированно);
- правила подбора: формулировать запросы терминологией ГЭСН (разборка, а не
  демонтаж; устройство, а не монтаж — где уместно); проверять состав работ
  (Content) перед выбором; учитывать, что часть позиций КП может входить
  в состав другой нормы (подступенники в норме ступеней) — тогда помечать
  как `included_in` с указанием куда; позиции вне ГЭСН (логистика, чертежи,
  диагностика, аренда) помечать `out_of_scope`, не подгонять ближайшую норму;
- формат ответа по каждой позиции — строгий JSON:

```json
{
  "item_no": "1.7",
  "status": "matched | uncertain | out_of_scope | included_in",
  "candidates": [
    {"base_type": "ГЭСН", "code": "15-01-038-01", "confidence": "high|medium|low",
     "reasoning": "почему эта норма; что сверено по составу работ",
     "unit_mismatch": {"quote_unit": "м.п.", "norm_unit": "100 м2",
       "needs": "ширина ступени для пересчёта"} }
  ],
  "notes": "включая сомнения и что уточнить у пользователя"
}
```

Технические требования к пайплайну:
- позиции обрабатывать батчами (по 5–8 за вызов), а не по одной — контекст
  ведомости общий, дешевле и связнее;
- лимит tool-вызовов на батч (например 25) и общий таймаут — защита от
  зацикливания; при обрыве батча позиции из него получают `status: "error"`,
  прогон продолжается;
- **начинать с одного батча, а не со всей ведомости.** Первый контакт модели
  с реальным промтом почти наверняка потребует итераций (формулировки, формат
  ответа, характер использования инструментов). Отладить на 5–8 позициях листа
  «Гранит», добиться стабильного валидного JSON — и только потом запускать
  полный прогон. Дешевле по токенам и быстрее по циклу правок;
- лог `runs/<timestamp>.jsonl` вести **с самого первого вызова**, включая
  отладочные прогоны одного батча: если модель начнёт подбирать странно,
  история диалога — единственный способ понять почему;
- весь диалог с моделью (запросы, tool-вызовы, ответы) писать в лог-файл
  прогона `runs/<timestamp>.jsonl` — для разбора ошибок подбора;
- ответы модели парсить устойчиво (JSON может прийти в ```json-обёртке).

## 5. Компонент 3: Экран подтверждения (UI, 5-й экран)

Таблица позиций ведомости, по каждой:
- статус цветом (matched / uncertain / out_of_scope / included_in / error);
- выбранный кандидат + альтернативы из candidates выпадашкой;
- reasoning — видимым текстом (не тултипом): это главная ценность против
  ручного подбора;
- при unit_mismatch — поле ввода недостающего размера (ширина, глубина...)
  и формула пересчёта объёма; без ввода позиция не считается;
- кнопки: подтвердить / заменить норму (открывает существующий поиск) /
  исключить;
- подтверждённые позиции уходят в существующий движок расчёта; итог —
  многопозиционная смета (см. п. 6) с экспортом.

Для сценария Б дополнительно в каждой строке: цена КП против нормативной,
разница в % и рублях; итоговая сводка по разделам (Работы/Материалы) и
общий вердикт (сумма КП vs сумма норматива по сопоставленным позициям,
отдельно — сумма несопоставленных).

## 6. Многопозиционная смета (минимум)

Этап 4 считал одну позицию. Здесь нужен контейнер: массив подтверждённых
позиций → расчёт каждой существующим движком → свод (ПЗ/ФОТ/НР/СП/НДС
по позициям и итогом) → экспорт в один xlsx (лист-свод + листы позиций или
свод с разбивкой — на усмотрение исполнителя, но итоги обязательны).
Сохранение смет между сессиями по-прежнему вне скоупа (v2) — прогон живёт
в памяти/в файле экспорта.

## 7. Приёмка

Приёмочный файл — `Гранит.xlsx` (положить в `data/`, в git не коммитить,
как остальные данные). Критерии:

1. Парсер извлекает оба листа полностью, все позиции с номерами/ед./кол-вом,
   секции и итоги распознаны (юнит-тест с точными числами).
2. Прогон сценария Б по листу «Гранит» завершается без ручного вмешательства;
   каждая из 14 позиций работ получает status и (для matched/uncertain)
   хотя бы одного кандидата с reasoning.
3. Контрольные позиции (сверка с ручным разбором, задокументирован в истории
   проекта): «Монтаж ступеней» (1.7) → среди кандидатов есть
   `ГЭСН 15-01-038-01` с unit_mismatch (м.п./100 м2); «Демонтаж металлических
   ограждений» (1.3) → `ГЭСНр 68-03-004-01`; «Сверление отверстий» (1.12) →
   серия `ГЭСН 46-03-001-XX`; «Транспортные расходы» (1.13) и «Подготовка
   чертежей» (1.1) → `out_of_scope`. Несовпадение с этим списком — не
   автоматический провал (модель может найти лучше), но каждое отклонение
   разобрать в отчёте руками.
4. Ни одна позиция не «посчитана молча» с несовпадающими единицами —
   unit_mismatch всегда требует ввода.
5. Экран подтверждения: замена кандидата и исключение позиции работают;
   после подтверждения считается свод и выгружается xlsx.
6. `npm test` зелёный, сверка эталона 45/45 не тронута; при отсутствии
   `AI_API_KEY` всё, что было до Этапа 5, работает как раньше.
7. В отчёте контрольной точки: полный вывод прогона по «Граниту» (таблица
   статусов), расход токенов справочно и стоимость прогона в рублях
   по разнице баланса кабинета (см. раздел 2).

## 8. Явно вне скоупа Этапа 5

- Автоприменение поправочных коэффициентов (по-прежнему v2; но если модель
  в reasoning подсказывает применимый коэффициент по `<Applys>`-привязкам —
  это плюс, не требование).
- Обучение/дообучение чего-либо; только промтинг + tool use.
- Пакетная обработка нескольких ведомостей, очереди, фоновые задания.
- Деплой и многопользовательский доступ.
- Сохранение смет в БД.

## 9. Порядок работы

Как в прошлых этапах: сначала проверка входных данных (структура `Гранит.xlsx`
по факту) и **smoke-тест tool use через neuroapi** (шаг 0 из раздела 2 — с его
результата начинается первый отчёт), контрольная точка после парсера
(компонент 1 отдельно, до подключения модели), затем пайплайн, затем UI.
Отчёты с дословными выводами, не пересобранными вручную.
