# Калькулятор ГЭСН-2022

Сметный калькулятор на базе ФСНБ-2022 (доп. №18), ресурсно-индексный метод
по Методике 421/пр. Считает нормативную стоимость одной расценки в текущих
ценах и сравнивает её с рыночной ценой из КП подрядчика.

Полное ТЗ — [`ТЗ_смета_ГЭСН2022.md`](ТЗ_смета_ГЭСН2022.md), оно является
источником истины по схеме БД, маппингу импорта и формулам расчёта.

## Стек

- Node.js 20+ (разработка на 24.16), без TypeScript и без фронтенд-сборки
- `better-sqlite3` — БД `gesn2022.db`
- `saxes` — потоковый парсинг XML (файлы до 84 МБ, DOM не используется)
- `exceljs` — чтение сплит-формы (281 тыс. строк, streaming) и экспорт
- `express` — минимальный веб-интерфейс (этап 4)

## Структура

```
src/
  db/        schema.sql, подключение к SQLite
  import/    импортёры: ГЭСН, ФСБЦ, ТГ, коэффициенты, сплит-форма, нормативы НР/СП
  engine/    calc-position.js — расчёт одной позиции (раздел 5 ТЗ)
  verify/    verify-etalon.js — сверка с эталонной ЛСР (раздел 8 ТЗ)
  web/       express-сервер + статический интерфейс
test/        юнит-тесты (node --test)
data/        исходные XML/XLSX — НЕ в git (171 МБ)
work_type_norms.csv   нормативы НР (812/пр) и СП (774/пр), 136 строк — в git
```

## Данные

Исходные файлы лежат в `data/` и не версионируются. Требуемый состав:

- 5 баз норм: `ГЭСН.xml`, `ГЭСНм.xml`, `ГЭСНмр.xml`, `ГЭСНп.xml`, `ГЭСНр.xml`
- 2 базы цен: `ФСБЦ_Мат&Оборуд.xml`, `ФСБЦ_Маш.xml`
- 2 файла техгрупп: `База ТГ.xml`, `Ключи перехода ТГ.xml`
- поправочные коэффициенты (доп. 18)
- сплит-форма региона на квартал (обновляется ежеквартально)
- эталонная ЛСР для верификации

Все XML — UTF-8 с BOM.

## Команды

```
npm run import                  # полный импорт справочников в gesn2022.db (~11 с)
npm run import:split -- <xlsx>  # импорт новой сплит-формы (ежеквартально)
npm run report                  # отчёт о содержимом БД после импорта
npm test                        # юнит-тесты расчёта (этап 2)
```

## Расхождения ТЗ и фактических данных

Проверка файлов до написания импортёров выявила четыре места, где данные не
укладываются в схему раздела 3 ТЗ. Все решения — аддитивные: колонки ТЗ
сохранены, добавлены только поля, без которых информация терялась бы молча.

1. **Несколько `<ReasonItem>` в одной норме** — 4 033 нормы (12 в ГЭСН,
   4 021 в ГЭСНм) допускают несколько видов работ, то есть несколько пар
   нормативов НР/СП. В `works.nr_code`/`sp_code` пишется первый вариант,
   все — в `work_nrsp_items`, у нормы поднимается `works.nrsp_ambiguous`.
   Движок обязан требовать выбор, а не молча брать первый.

2. **`Quantity="П"`** («по проекту») вместо числа — 14 474 ресурса, из них
   12 478 абстрактных. Схема ТЗ требует `quantity REAL NOT NULL`, что делает
   импорт невозможным. `NOT NULL` снят, добавлена `quantity_note='П'`.
   Расход таких ресурсов задаёт пользователь.

3. **Пункты 104.1 и 104.2** используются в 249 нормах ГЭСНмр, но в
   `work_type_norms.csv` им соответствуют строки `104` и `104доп`, и какая
   из них какому пункту отвечает — из данных не следует. Норматив для этих
   норм не подставляется автоматически.

4. **106 строк сплит-формы** не позволяют вычислить текущую цену (91 строка
   имеет базисную цену без индекса, 15 — пустые). Такие ресурсы помечаются,
   а не считаются по базисной цене.

Дополнительно: 1 653 из 6 710 технологических групп объявлены в `База ТГ.xml`
пустыми, из-за чего у 3 859 норм список кандидатов на основной материал пуст —
код придётся вводить вручную.
