# Калькулятор ГЭСН-2022

Сметный калькулятор на базе ФСНБ-2022 (доп. №18), ресурсно-индексный метод
по Методике 421/пр. Считает нормативную стоимость одной расценки в текущих
ценах и сравнивает её с рыночной ценой из КП подрядчика.

Полное ТЗ — [`ТЗ_смета_ГЭСН2022.md`](ТЗ_смета_ГЭСН2022.md), оно является
источником истины по схеме БД, маппингу импорта и формулам расчёта.

## Стек

- Node.js 20+ (разработка на 24.16), без TypeScript и без фронтенд-сборки
- `better-sqlite3` — БД `gesn2022.db`
- `saxes` — потоковый парсинг XML (файлы до 84 МБ, DOM не используется)
- `exceljs` — чтение сплит-формы (281 тыс. строк, streaming) и экспорт
- `express` — минимальный веб-интерфейс (этап 4)

## Структура

```
src/
  db/        schema.sql, подключение к SQLite
  import/    импортёры: ГЭСН, ФСБЦ, ТГ, коэффициенты, сплит-форма, нормативы НР/СП
  engine/    calc-position.js — расчёт одной позиции (раздел 5 ТЗ)
  verify/    verify-etalon.js — сверка с эталонной ЛСР (раздел 8 ТЗ)
  web/       express-сервер + статический интерфейс
test/        юнит-тесты (node --test)
data/        исходные XML/XLSX — НЕ в git (171 МБ)
work_type_norms.csv   нормативы НР (812/пр) и СП (774/пр), 136 строк — в git
```

## Данные

Исходные файлы лежат в `data/` и не версионируются. Требуемый состав:

- 5 баз норм: `ГЭСН.xml`, `ГЭСНм.xml`, `ГЭСНмр.xml`, `ГЭСНп.xml`, `ГЭСНр.xml`
- 2 базы цен: `ФСБЦ_Мат&Оборуд.xml`, `ФСБЦ_Маш.xml`
- 2 файла техгрупп: `База ТГ.xml`, `Ключи перехода ТГ.xml`
- поправочные коэффициенты (доп. 18)
- сплит-форма региона на квартал (обновляется ежеквартально)
- эталонная ЛСР для верификации

Все XML — UTF-8 с BOM.

## Команды

```

npm run index                   # перестроить поисковый индекс (входит в import)
npm run dev                     # веб-интерфейс на http://localhost:3000

```

## Веб-интерфейс

Четыре экрана-состояния: выбор или загрузка сплит-формы → поиск расценки
(по коду и полнотекстовый по названию) → карточка расчёта (объём, основной
материал из технологической группы, НДС, территория, коэффициент к норме,
рыночная цена из КП) → результат с построчной разбивкой, сравнением
«норматив против рынка», кодами для переноса в Гранд-Смету и выгрузкой
в Excel.

API: `GET /api/periods`, `POST /api/periods` (загрузка xlsx),
`GET /api/search?q=`, `GET /api/work/:base/:code`, `POST /api/calc`,
`POST /api/export`.

## Поиск расценок

Поиск по названию рассчитан на то, как человек формулирует запрос, а не на
дословную цитату из наименования нормы:

- **основы слов вместо точных форм** — стеммер Snowball для русского
  (`src/search/stem-ru.js`, без зависимостей), плюс правило беглой гласной
  («стяжки» ↔ «стяжек», «перегородки» ↔ «перегородок»), которое Snowball
  не покрывает;
- **поиск по началу слова**, а не по точному совпадению;
- **опечатки** — словарь индексных терминов `search_vocab`, кандидаты
  подбираются через триграммный индекс FTS5 `vocab_trigram`, отбираются по
  расстоянию Левенштейна ≤ 2 («металических» → «металлическ»);
- **синонимы** — таблица `search_synonyms`, заполняется из
  `src/search/synonyms.js` и пополняется вручную:
  `INSERT INTO search_synonyms (term, alias, source) VALUES ('разборк', 'демонтаж', 'manual');`
  Термины хранятся уже в виде основ.

В индекс кладутся и основа, и исходное слово: Snowball местами срезает
лишнее («гранит» → «гран»), и наличие обеих форм закрывает промахи в обе
стороны. В запрос идёт только основа — она всегда является префиксом слова,
поэтому второй вариант ничего не добавил бы к отзыву, зато исказил бы bm25.
Порядок выдачи поправляется переранжированием: нормы, где встретилось само
слово запроса, поднимаются над теми, что совпали лишь по укороченной основе.

Ответ `/api/search` содержит поле `terms` с разбором запроса (что во что
стеммировано, что исправлено, какие синонимы добавлены) — интерфейс
показывает это под строкой поиска.

## Расхождения ТЗ и фактических данных

Проверка файлов до написания импортёров выявила четыре места, где данные не
укладываются в схему раздела 3 ТЗ. Все решения — аддитивные: колонки ТЗ
сохранены, добавлены только поля, без которых информация терялась бы молча.

1. **Несколько `<ReasonItem>` в одной норме** — 4 033 нормы (12 в ГЭСН,
   4 021 в ГЭСНм) допускают несколько видов работ, то есть несколько пар
   нормативов НР/СП. В `works.nr_code`/`sp_code` пишется первый вариант,
   все — в `work_nrsp_items`, у нормы поднимается `works.nrsp_ambiguous`.
   Движок обязан требовать выбор, а не молча брать первый.

2. **`Quantity="П"`** («по проекту») вместо числа — 14 474 ресурса, из них
   12 478 абстрактных. Схема ТЗ требует `quantity REAL NOT NULL`, что делает
   импорт невозможным. `NOT NULL` снят, добавлена `quantity_note='П'`.
   Расход таких ресурсов задаёт пользователь.

3. **Пункты 104.1 и 104.2** используются в 249 нормах ГЭСНмр, а в
   `work_type_norms.csv` им отвечают строки `104` и `104доп`. Сопоставление
   установлено по Отделу: все 218 норм с 104.1 лежат в Отделах 01-04, все
   31 норма с 104.2 — в Отделах 05-06, нарушений границы ноль, что совпадает
   с формулировками CSV («кроме отделов 5,6» / «отделы 5,6»). Задано таблицей
   `ITEM_NO_ALIASES` в `import-norms-csv.js`; норматив НР/СП подбирается для
   100% норм всех пяти баз.

4. **106 строк сплит-формы** не позволяют вычислить текущую цену (91 строка
   имеет базисную цену без индекса, 15 — пустые). Такие ресурсы помечаются,
   а не считаются по базисной цене.

5. **Единицы измерения внутри технологической группы не совпадают с единицей
   абстрактного ресурса** — у 32.4% пар «абстрактный ресурс × кандидат»
   (норма в тоннах, материал в штуках или килограммах и т.п.). Пересчёт
   единиц система не выполняет: при несовпадении расход обязан задать
   пользователь уже в единицах выбранной марки, иначе строка не считается
   и поднимается флаг `единицы_измерения_не_совпадают`.

Дополнительно: 1 653 из 6 710 технологических групп объявлены в `База ТГ.xml`
пустыми, из-за чего у 3 859 норм список кандидатов на основной материал пуст —
код придётся вводить вручную.

## AI-слой (Этап 5)

Подбор норм ГЭСН к ведомости подрядчика через внешний API. Провайдер —
neuroapi.host (OpenAI-совместимый агрегатор), SDK `openai` с кастомным
`baseURL`, конфиг в `.env` (`AI_API_KEY`, `AI_BASE_URL`, `AI_MODEL`). Без
ключа AI-функции отключены, всё остальное работает офлайн.

### Ограничение провайдера: prompt caching недоступен

Проверено на шаге 0 и подтверждено измерением: **neuroapi не прокидывает
prompt caching к Anthropic** — заголовок `cache_control` игнорируется,
`cached_tokens` остаётся 0 на повторном вызове. Это закрывает главный рычаг
экономии на входных токенах: в пайплайне tool use ~80–97% расхода — вход,
и он растёт с каждым раундом из-за пересылки истории диалога. Кэш снял бы
эту стоимость на повторяющейся части (системный промт + контекст ведомости),
но через этот прокси он невозможен.

Что сделано взамен (структурные меры): секции «Материалы»/«УСН» не идут в
пайплайн подбора норм (классифицируются парсером); `get_norm_details`
по умолчанию отдаёт сводку вместо полного состава (−67–69% на ответ,
измерено напрямую); ретраи на транзиентных 500 прокси.

Счётчики токенов у прокси недостоверны (для одной модели 4 399 prompt на
короткую фразу, у другой `cached_tokens` > `prompt_tokens`), поэтому
**стоимость меряется по разнице баланса кабинета**, а токены — справочно.

### Выбор моделей

`AI_MODEL` основная + `AI_MODEL_FALLBACK` запасные (через запятую). При
полном отказе основной у агрегатора батч переигрывается на запасной;
переключение видно по каждой позиции (`result.model`, `result.model_fallback`),
а не только в логе.

Подбор моделей проверен прогонами листа «Гранит» (14 работ), стоимость —
по балансу кабинета:

| Модель | matched | контр. точки | галлюц. | дефект | $/прогон | вендор |
|---|---|---|---|---|---|---|
| **claude-sonnet-5** (осн.) | 8 | 5/5 | 0 | нет | 0.68 | Anthropic |
| **gpt-5.6-terra** (зап.) | 1–2 | 5/5 | 0 | переосторожен (uncertain, но кандидаты верны) | 0.25 | OpenAI |
| haiku-4.5 | 5 | 4/5 | 0 | якорение на одном коде | 0.27 | Anthropic |
| sonnet-4-6 | 3 | 4/5 | 0 | неверный out_of_scope на реальных работах | 0.58 | Anthropic |

Запасная — **gpt-5.6-terra**: главный аргумент независимость от вендора
(отказ Anthropic upstream кладёт всю Claude-линейку разом), плюс дешевле
sonnet-5 и без поведенческих дефектов haiku/4.6. Её переосторожность
(uncertain вместо matched) для запасной модели приемлема: позиции помечены
⇄ЗАПАСНАЯ и идут на ручную проверку, а сами кандидаты верны.

**Накрутка входных токенов у GPT — опровергнута для gpt-5.6-terra.** На
шаге 0 `gpt-5.4` показывал раздутый usage (13 836 токенов там, где Claude
укладывался в ~400), но по БАЛАНСУ Terra стоил $0.25 — дешевле sonnet-5;
раздутые счётчики шага 0 были артефактом счётчика конкретной модели, а не
реальным расходом. Ещё одно подтверждение, что мерить надо по балансу.
